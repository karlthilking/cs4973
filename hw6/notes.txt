
Questions:
- Should a cache line be marked modified if it is first loaded in because a 
  write is being performed?
  - Answer: cache is a subset of higher memory, if a data block is needed for
    writing and is missed in L1, it is copied into cache and written to
    atomically -> modified bit is set, modified bit must always be set when
    a copy of a cache line doesn't reflect the cache line in higher memory,
    so when a line is first copied in, it must marked modified (it could also
    write through to higher memory, but that would be inefficient)
- Do cpu caches ever try to remove cache lines in advance to maintain 
  a threshold of availability
    - Answer: if a lower level cache is caching data from higher memory that
      is then swapped out to disk, the contents of the cache for the swapped
      page will be cleared, otherwise, cache lines are only cleared/removed
      when a cache line needs to be evicted
- Should a cache line be aligned to the data block size (e.g. multiple of 64)
  or simply aligned to the word size (multiple of 4 bytes)
    - i.e. if accessing a word at virtual address 0x00000048 (72) should the 
      cache line start at 0x00000040 (64; aligned to size of cache line) or 
      0x00000048 (72; word aligned address)
    - Answer: the tag of the line will be the higher order bits of an address
      so left shifting the tag to form an address creates an address aligned
      to the size of a block (i.e. for 64 bytes line TAG << (index+offset)
      will be a cache line size aligned address)

Direct-mapped Cache
- low bits = index into cache, high bits = tag bits to avoid collision
- example parameters: 32 bit address, 64 byte blocks, 32KB cache, 512 blocks
    - 17 bits to address tag, 9 to address one block, 6 bits to address byte in block
- another example: 64kb cache, 4 byte (word) blocks, 32 bit address (assume
  only tag bits and a valid bit)
    - 64KB = 2^16, 4 bytes = 2^2 -> 2^14 total blocks (16K)
    - 16 tag bits, 14 index bits, 2 offset bits
    - each block has 32 bits (word) of data + tag bits + valid bit = 49 bits
    - data cache size in bits = (bits/blocks) * blocks = 49 * 2^14 = 784 Kbits

Set-associative Cache
- an N-way associative cache has N cache entries where it can place a block,
  and has a number of sets each containing N blocks
- each blocks includes an address tag
- address decomposes into -> | tag | index | offset |
- Example 32KB L1D cache, 64 byte block size, 8 way associative
    - (2^15)/(2^6) data blocks = 512 blocks -> 64 sets of 8 blocks
    - index = address % number of sets
- Example 4K blocks (2^12), 4 word block size (16 bytes), 32 bit address,
  4 way associative
    - total size: 2^12 * 2^4 bytes = 2^6 KB = 64KB
    - total sets: number of blocks / associativity = 1024
    - 10 bits needed to index into a set, 4 bits to select byte in block -> 
      18 bits for tag
    - total tag bits = tag bits * associativity * sets = 18*4*1024 = 72Kbits

Fully-associative Cache
- no index bits, entire address ^ (offset bit mask) is used
    
Bit Representation
- Tag Bits
    - higher order bits of address
    - for fully-associative cache, the tag is the full upper portion of the
      address bits except for the offset (e.g. for 64 byte block, 6 bits for
      offset, rest or for tag) because there are not index bits for a fully
      associative cache (entire cache is searched for a given address)
    - for set-associative cache, the tag is the upper portion of the address
      that remains after using the lowest order bits for offset, and the next
      lowest order bits for the index
        - set-associative example: 64 byte block, 32kb cache, 8-way:
            - assume 32 bit address
            - 6 bits are used for offset to address each byte in a block
            - 32kb / 64b = (2^15)/(2^6) = 2^9 = 512 individual blocks
            - each set has 8 blocks, 512 blocks -> 64 sets -> 6 index bits
            - then 20 bits are used for tag
    - for direct-mapped cache, tag bits are higher order bits leftover after
      log_2(number of blocks) bits for index and log_2(block size) offset bits
- Index Bits
    - for fully-associative cache, there are no index bits because a block can
      be placed anywhere in the cache
    - for set-associative cache, the number of index bits = log_2(# of sets)
    - for direct-mapped cache, the number of index bits = log_2(# of blocks)
- Valid Bit
    - 0: invalid, 1: valid
    - a cache block is invalid if a process is ran following a context switch
      and the block was used for the previous running process
    - a cache block is also invalid when a processor first begins running,
      the data is the cache is junk, so each valid bit is cleared
    - a block is also invalid if another copy of the block in a different
      private cache (shared block) was modified, then, the entry must be
      invalidated because it does not reflect the most recently updated copy
      of that data
- Modified Bit
    - set whenever a cache block is written to, used to maintain cache
      coherency, also used to avoid unecessary write backs when evicting a line
- Use/Reference bit (useful in LRU eviction)
    - set whenever a cache entry is referenced, cleared whenever an entry needs
      to be eviceted from the cache; when eviction takes place, first entry
      without the use bit unset in evicted

Write-back policy:
    - avoid writing through to main memory whenever a cache entry is updated,
      instead only when a cache block is evicted, check the modified bit; if
      the modified bit is set the entry needs to be written to higher memory
      now

Cache Miss Control Flow:
    - send address to higher memory
    - write cache entry, put data into data portion, write tag bits, set valid

MSI Protocol:
    - modified, shared and invalid bits
    - set modified bit when performing a write
    - shared bit indicates that a block is shared and valid
    - invalid bit is set if the block was initially shared, but another cpu
      modified its copy of the block

Physically vs Virtually Indexed Cache:
    - with physically indexed cache, the TLB or page table provides the 
      physical address which is used to index into the cache
    - with virtual indexed cache, the generated virtual address used to read
      or write can be used directly to index the cache; the address only needs
      to be translated when there is cache miss to obtain the data from memory
      - Issue with virtual indexed cache:
        - two processes could be sharing memory, but because the cache is
          virtually indexed, there could be two seperate copies in a cache
          and could possibly become inconsistent
        - Solution: Virtual index with physical tag
            - TLB and cache are accessed in parallel when a virtual address
              is generated; cache is indexed with virtual address index bits,
              TLB obtains physical tag for the virtual address, then physical
              tag is compared with entry in cache that was indexed with the
              virtual address to validate the cache entry
